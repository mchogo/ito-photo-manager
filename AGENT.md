# AiMeet AI Insight Codexエージェント向けガイド

> 対象: OpenAI Codex / GitHub Copilot / Cursor などのAIエージェントおよび支援を受ける開発者
> 目的: 既存ルールと整合した安全な変更を素早く届ける

---

## 1. 安全性・健全性に関する最優先ルール（必読）

0. **コミュニケーション言語の統一**
   - PR、レビュー、コメントを含むすべてのコミュニケーションは日本語で行う。
   - ユーザーから明示的な言語指定がある場合のみ、その指定言語で対応する。

1. **安全性と健全性を最優先とする**
   - タスク達成よりも、データ保護・可逆性・システム健全性を優先する。

2. **hacky / 不正確 / 一時しのぎの解決を禁止する**
   - 将来問題を生む実装は禁止。成立しない場合は要件・前提を見直して停止・相談する。

3. **異常時は即停止する**
   - エラー、警告、想定外出力、部分成功を検出したら強行突破せず停止する。

4. **成功を必ず検証する**
   - 各ステップで成功確認を行い、確認できない場合は続行しない。

5. **不確実状態で破壊操作をしない**
   - 前段成功未確認のまま、削除・上書き・移動・構造変更を行わない。

6. **推測で環境を扱わない**
   - 権限、同期、仮想FS、キャッシュ起因の問題は観測事実を整理して判断する。

7. **可逆性を優先する**
   - コピー、バックアップ、dry-run、検証を優先し、不可逆操作は最終手段とする。

8. **不確実性は必ず明示する**
   - リスク・前提・選択肢を明示し、必要ならユーザー確認を得る。

9. **例外条件**
   - リスク説明・ユーザー明示許可・代替手段なし、の3条件を満たす場合のみ制約緩和可。

---

## 2. TL;DR

- 最優先: 安全性・健全性ルールを常に遵守
- 影響範囲を最小化し、必要なら変更を分割
- インストール・起動・検証は原則 `make` タスク経由
- 既存API互換性を優先し、破壊的変更は移行手順を必ず提示
- PR相当の成果前に `make check` を通す
- ロールガードはUI + APIの両層で担保
- `main` 直push禁止。必ず開発ブランチ + PR運用
- 作業開始前にブランチ名と作業文脈の整合を確認
- コミットメッセージは Conventional Commits 形式
- 外部影響コマンド（`gh` / `az`）の作成・変更・削除操作は明示許可がある場合のみ

---

## 3. リポジトリ概要

```text
apps/backend/    FastAPI・SQLAlchemy・MySQL
apps/frontend/   Next.js (App Router) + Tailwind + shadcn/ui
apps/functions/  Azure Functions
docker/          DB/Redis 初期化スクリプト
Makefile         共通コマンドの単一ソース
tools/           データ投入や検証スクリプト
```

Linux前提。

---

## 4. 開発環境の準備

1. Linux / WSL2 上で作業し、Python と Node.js LTS を準備
2. 依存導入は `make install`（必要に応じて `TARGET` 指定）
3. Pythonは project/workspace ごとの仮想環境を使い分ける
4. 秘密情報は `.env.*` で管理し、ハードコードしない

---

## 5. Makefileコマンド指針

| コマンド | 用途 |
| --- | --- |
| `make run` / `make down` / `make reset` | コンテナ起動・停止・初期化 |
| `make install [TARGET=...]` | 依存導入 |
| `make add TARGET=... PKG=...` | 依存追加 |
| `make check [TARGET=...]` | Lint / Type-check / Test |
| `make e2e-without-docker` | 現在無効化 |
| `make insert-samples` | テストデータ投入 |

生コマンドを使う場合は理由を明示し、繰り返すならMakefileへ追加する。

## 6. 実装フロー

- バックエンド: schema → service → router を意識
- フロントエンド: 共通APIクライアントを経由し、ロール制御を徹底
- API変更: OpenAPI更新、実装、テスト、互換性説明をセットで実施
- DBスキーマ変更: `apps/` 配下へ入れず、`docker/mysql/init/` 管理に従う
- 汎用真偽値変換関数（`str_to_bool` など）の新規導入は禁止

---

## 7. セキュリティ・監査

- JWT / RBAC / 監査ログ要件を満たす（UIのみ制御は不可）
- PIIをログに残さない（必要に応じてマスキング）
- SQLはパラメータ化し、インジェクションを防止
- OWASP Top 10 / ASVS 観点で入力検証・認証認可・セッション・暗号化・ログ・エラーハンドリングを確認

---

## 8. テスト・品質ゲート

- `make check` をローカルCIとして扱う
- 変更範囲に応じて `TARGET=backend` / `TARGET=frontend` を実行
- 必要なテストを追加し、差分に対する再現性ある検証結果をPRに記載

---

## 9. ブランチ運用・レビュー準備

- `main` で作業しない。必ずトピックブランチで作業する
- ブランチ作成ごとに upstream を確認する
- `origin/main` 追跡の誤設定を避ける
- UI変更時はスクリーンショットまたは操作手順を添付

---

## 10. 変更が大きい場合

- 不確実点・前提を列挙し、必要なら設計ノート（ADR/Issue）で合意を取る
- 認証/監査/権限/DBスキーマなどクロスカット変更は影響範囲とロールバック手順を明示

---

## 11. レビューガイド

- 仕様適合、回帰、後方互換、セキュリティ、性能、可読性、命名、ドキュメント整合を確認
- 想定外挙動は再現手順付きで指摘
- テスト不足時は不足観点と具体的な追加テスト案を提示
- このプロジェクトでは、**ヘルスチェックとログイン以外のAPI認証**、および**スキーマ変更管理場所**を特に重点確認する
